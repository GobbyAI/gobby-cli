# Architecture Overview

## Overview

The repository is partitioned as independent subsystem layers rather than a tightly coupled crate graph: the supplied dependency map says there are no cross-subsystem dependency edges. The foundation is `crates/gcore`, which centralizes shared configuration, service provisioning, database/vector adapters, setup validation, secrets, project discovery, AI routing, indexing/search primitives, and degradation/CLI contracts. Product-facing layers build on those shared concepts without owning them globally: `crates/gcode` owns code indexing, import/call resolution, PostgreSQL persistence, FalkorDB/Qdrant projection, code search, setup/freshness, and Codewiki-style output; `crates/gwiki` owns the local-first wiki CLI, raw-source ingestion, wiki indexing/search, provenance/audit/health/trust, and synthesis/media/text workflows.

The boundary is intentional: `gcore` supplies reusable infrastructure and degradation semantics while keeping domain-specific graph names, collection names, and consumer query behavior out of the shared layer. That lets `gcode` and `gwiki` own their domain contracts and data models independently, while `ghook` owns only host-agent integration: it captures hook invocations, writes durable local inbox envelopes before network delivery, best-effort posts to the daemon, and maps daemon responses or transport failures back into the stdout, stderr, and exit-code shape expected by the invoking CLI. `docs/evidence` and `scripts` sit beside the product crates as operational support: evidence archives preserve verification context and results, while `scripts/verify.sh` standardizes workspace quality gates.

The required services are the persistence and indexing infrastructure that the product surfaces are defined around, especially PostgreSQL for `gcode` source indexing and the graph/vector backends used when `gcode` projects relationship and embedding data into FalkorDB and Qdrant. Some infrastructure is required for the full product but has degraded command behavior when unavailable: the daemon path is degraded by `ghook` into durable local capture plus best-effort delivery, with transport failures converted into the invoking CLI’s stdout, stderr, and exit status rather than erasing the hook event. AI routing is also treated as degradable infrastructure: `gcore` owns AI capability routing/degradation contracts, `gwiki` uses AI-assisted synthesis/media/text processing, and `docs/evidence` records AI routing/degradation status for later review.

The runtime model therefore has two modes. In standalone mode, `gcode`, `gwiki`, verification scripts, and evidence workflows operate through their local CLI/application contracts and must respect their own data ownership: code-index data, import roots, graph/vector projections, wiki raw sources, provenance, audit records, and evidence artifacts stay with their owning subsystem. In daemon-attached mode, `ghook` becomes the bridge from host agent CLIs into Gobby, but it still does not own product data; it owns envelopes, delivery durability, and CLI-compatible response translation, while daemon-consumable contracts remain defined by the product surfaces that expose them.
